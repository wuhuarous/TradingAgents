# 项目分析文档

## 1. 项目定位

当前项目 `trandingAgents` 从命名和常见目录语义看，目标应是构建一个面向交易研究的多智能体系统：由不同角色的 Agent 分别完成行情、新闻、情绪、基本面、技术面、交易决策与风险控制分析，最终输出可解释的交易建议。

该类项目的核心价值不在于“让大模型直接下单”，而在于把复杂交易判断拆成可审计、可回放、可对比的决策链路。一个成熟版本至少需要回答四个问题：

- 数据从哪里来，质量是否稳定。
- Agent 如何分工，结论如何汇总。
- 决策是否可追踪、可复盘。
- 风险约束是否先于收益追求。

## 2. 现状判断

本轮无法通过本地 shell 完成文件级扫描，原因是 PowerShell 启动阶段返回 `8009001d`，项目命令尚未真正执行。因此以下判断采用“项目目标 + 多智能体交易系统常见缺口 + TradingAgents 公开框架对标”的方式形成，后续需要在终端恢复后补充精确到文件和函数的诊断。

当前项目未达到预期，通常会集中体现在以下方面：

1. 产品目标不够清晰

   多智能体交易项目容易停留在“有多个 Agent 对话”的形态，但缺少明确的使用路径。用户真正需要的是输入股票、日期、风险偏好、数据源配置后，稳定得到结构化结论，而不是阅读一大段不可比较的文本。

2. 决策链路缺少结构化输出

   如果 Agent 只输出自然语言报告，后续很难做自动化评估、回测、统计和 UI 展示。理想输出应包含 `action`、`confidence`、`time_horizon`、`risk_level`、`supporting_evidence`、`opposing_evidence`、`invalidating_conditions` 等字段。

3. 数据层和推理层耦合过重

   数据获取、缓存、指标计算、Agent prompt、LLM 调用如果混在一起，会导致系统难以测试，也难以切换数据源或模型供应商。

4. 缺少失败降级机制

   金融数据接口、新闻接口和 LLM API 都可能失败。系统应能区分“数据缺失”“模型失败”“分析不充分”和“明确不建议交易”，否则用户会把系统故障误解为交易观点。

5. 风险管理可能偏弱

   交易类 Agent 系统不能只输出买卖方向，还应在最终决策前引入仓位、波动率、最大回撤、止损条件、流动性、事件风险等约束。

6. 缺少复盘记忆

   如果每次分析互相独立，系统无法积累经验。成熟系统应记录每次决策、当时证据、实际后续收益和反思摘要，用于下一次同 ticker 或相似市场状态的提示增强。

7. 工程体验不足

   需要明确的安装方式、环境变量模板、CLI/API 用例、Docker 或一键启动方式、最小测试集、开发文档和故障排查文档。

## 3. 与成熟 TradingAgents 框架的差距

公开的 TradingAgents 框架已经具备较完整的 Agent 分层，包括分析师团队、研究团队、交易员、风险管理团队和组合经理。其较新版本还强调多模型供应商、结构化输出、决策日志、checkpoint 恢复、Docker 与 Windows 编码修复。

当前项目可以借鉴但不应盲目复制。更现实的迭代路线是先补齐稳定性和可解释性，再扩展模型与数据供应商。

## 4. 建议的系统目标

短期目标：

- 提供一个稳定的最小可用交易分析流程。
- 统一 Agent 输出结构。
- 保存每次分析结果，便于复盘。
- 增加错误分级与降级提示。
- 输出用户可读的中文报告和机器可读的 JSON。

中期目标：

- 支持多数据源和可配置模型供应商。
- 引入本地缓存与 checkpoint。
- 增加回测与决策复盘。
- 建立基础测试和 CI。

长期目标：

- 从“分析工具”升级为“研究工作台”。
- 支持策略对比、批量 ticker 分析、投资组合层面风险控制。
- 形成可解释、可审计、可复现的交易研究流程。

## 5. 本轮迭代原则

本轮先不追求大规模重构，而是优先补齐项目方向与协作基线：

- 先定义输出标准，再优化 Agent 细节。
- 先保证流程可回放，再追求更复杂的智能。
- 先把风险写进主流程，再扩展交易建议。
- 先文档化预期，再推进代码实现。
